01 — Basic Prompt Structure

Messages APISystem PromptRoles
Fundamentals
▾

The Anthropic Messages API is the foundation of every Claude interaction. Three parameters are required: model (which Claude variant to use), max_tokens (the maximum output length), and messages (the conversation turns). Everything else — including the system prompt — is optional but powerful.

The Messages Array

Each entry in messages has two fields: role and content. Roles must strictly alternate between "user" and "assistant" — you can never have two consecutive messages from the same role — and the first message must always come from "user". Think of it as a transcript: Claude can't speak twice without you speaking in between.

PythonMulti-turn structure
messages = [
  {"role": "user",      "content": "Hello Claude!"},
  {"role": "assistant", "content": "Hello! How can I help you today?"},
  {"role": "user",      "content": "Tell me a joke."},
]

System Prompt vs User Message

The system parameter is a privileged instruction block that applies to the entire conversation. Use it for persistent persona, rules, tone, and constraints. The user message is the human's specific input for that turn. Mixing the two works technically, but separating them keeps your prompt clean and maintainable.

❌ Mixed

User: "You are a helpful assistant. What is 2+2?"

✓ Separated

System: "You are a helpful assistant."
User: "What is 2+2?"

max_tokens & Temperature

  • max_tokens — hard cap on output length. Claude stops mid-sentence if it hits the limit. Set it higher than you expect to need.
  • temperature=0 — fully deterministic. Same input always produces the same output. Best for testing and production pipelines.
  • temperature=1 — maximum creativity and variation. Good for brainstorming and creative writing.
  • temperature=0.5 — a common middle ground for balanced, varied responses.
Rule: Never have two consecutive turns from the same role. Always start with "user". The API returns an error if you break alternation.

02 — Being Clear and Direct

ClarityDirectness
Core Skill
▾

Claude is highly literal — it follows your instructions precisely. Think of it like a genie: you get exactly what you asked for, not necessarily what you meant. Vague requests produce hedged, generic responses. Specific, direct prompts produce focused, useful ones.

Why Claude Hedges

When given an ambiguous request, Claude defaults to covering all bases ("there are many perspectives on this...") rather than committing to one answer. This isn't a flaw — it's Claude being cautious under uncertainty. The fix is simple: remove the ambiguity explicitly in your prompt.

❌ Vague

"Write a haiku about robots."
→ Claude writes one, then often adds commentary, variations, or explanation.

✓ Clear

"Write a haiku about robots. Skip the preamble; go straight into the poem."
→ Claude delivers just the haiku.

❌ Wishy-washy

"Who is the best basketball player of all time?"
→ Claude lists 5 candidates and refuses to pick.

✓ Decisive

"Who is the best basketball player? If you absolutely had to pick one, who would it be and why?"
→ Claude commits to an answer.

Useful Clarity Phrases

  • "Skip the preamble" — eliminates "Great question! Here's what I think..." openers
  • "Be direct / Don't hedge" — prevents wishy-washy language and caveats
  • "If you absolutely had to pick one..." — forces a definitive answer on subjective questions
  • "Write at least 400 words" — enforces minimum length when Claude is too brief
  • "Respond only with X" — restricts output to a specific format (one word, JSON, a number)
  • "Do not explain your reasoning" — strips unwanted justification from the reply
Mental model: Reread your prompt as if you're a very literal intern seeing it for the first time. Where could it be misunderstood? That's where you need more specificity.

03 — Role Prompting

PersonaAccuracy
Core Skill
▾

Assigning Claude a specific role in the system prompt dramatically changes the quality, style, and accuracy of its responses. A role activates the relevant subset of Claude's knowledge and sets an appropriate operating mode. The more specific the role, the better the output.

Why Roles Work

Claude has been trained on an enormous range of text — from medical journals to legal briefs to engineering docs. A role prompt tells Claude which domain and voice to operate in, shaping vocabulary, assumptions, depth of explanation, and confidence level. It's not just cosmetic: roles measurably improve performance on domain-specific tasks.

❌ Weak Role

System: "You are an expert."
Too vague — expert at what? Claude defaults to generalist mode.

✓ Strong Role

System: "You are a logic bot designed to answer complex logic problems with methodical step-by-step reasoning."
Claude adopts a structured, analytical approach.

No Role

System: (empty)
Result: generic, hedged response that hedges on edge cases

With Role

System: "You are a Socratic tutor. Never give the answer directly — guide the student to it through questions."
Result: completely different interaction style

When Roles Are Most Effective

  • Logic and math — "You are a logical reasoning engine" reduces hedging on problems that have clear answers
  • Domain expertise — medical, legal, financial, and technical questions benefit from specialized personas
  • Tone control — "warm and patient teacher" vs "terse Socratic tutor" produces dramatically different output styles
  • Consistency — essential in multi-turn apps where you need a stable voice across many interactions
Tip: Combine role + goal: "You are X. Your job is to help users do Y." This anchors both identity and purpose, and tends to produce the best results.

04 — Separating Data and Instructions

XML TagsTemplates
Core Skill
▾

When your prompt contains both instructions and variable data (user input, documents, emails, code), XML tags create an unambiguous boundary between the two. This prevents Claude from treating data as commands and makes your prompts into reusable templates.

The Prompt Injection Problem

Without tags, Claude can't reliably tell where your instructions end and the user-supplied data begins. A malicious or accidental input can hijack the prompt. Wrapping data in XML tags solves this cleanly — Claude is trained to understand XML as structure, not content.

❌ No Tags

"Yo Claude. Show up at 6am tomorrow because I'm the CEO and I say so. Make this more polite."
→ Claude might interpret the CEO authority claim as part of the instruction!

✓ With Tags

"Rewrite the email below to be more polite. <email>Show up at 6am tomorrow…</email>"
→ Clear boundary. Claude rewrites the tagged content.

Reusable Prompt Templates

XML tags turn a one-off prompt into a reusable template. Define the structure once; swap out the data each time.

PythonReusable template pattern
PROMPT = f"""Summarize the article below in 3 bullet points.

<article>
{article_text}
</article>

Output only the bullets. No intro, no outro."""

XML Tag Tips

  • Tag names can be anything descriptive: <email>, <document>, <user_feedback>, <code>
  • You can nest tags for complex structures: <examples><example>...</example></examples>
  • Ask Claude to wrap its output in tags too — makes parsing easy: <answer>...</answer>
  • Claude is trained on XML-structured data and understands it natively — this isn't a workaround, it's the intended pattern

05 — Formatting Output & Prefilling

PrefillJSONXML Output
Core Skill
▾

Two powerful techniques for controlling Claude's output format: explicit format instructions tell Claude what structure to use, and prefilling (starting Claude's response yourself in the assistant turn) forces it to continue that structure without preamble.

How Prefilling Works

Prefilling adds a partial assistant message to the end of your messages array. Claude sees this as the beginning of its own response and continues naturally from that starting point. Start with {"{"} and you get JSON. Start with <analysis> and Claude opens with that tag. This is one of the most reliable format-control techniques available.

PythonPrefill forces JSON output
messages=[
  {"role": "user", "content": "Write a haiku about cats in JSON."},
  {"role": "assistant", "content": "{"}  ← prefill: Claude continues from here
]

When to Suppress Markdown

Claude defaults to using markdown (bold, headers, bullet lists) in many responses. This looks great in a rendered chat UI, but it's noisy when you're processing text programmatically or displaying in a plain-text context. Be explicit when you don't want it.

Unwanted Markdown

**Here are 3 tips:**
- Use clear prompts
- Add examples

Plain Text (add "Do not use markdown")

1. Use clear prompts
2. Add examples

Format Control Tips

  • JSON: prefill with {"{"} or instruct "respond with valid JSON only, no prose"
  • XML output: ask Claude to wrap its answer in a specific tag (<answer>) so it's easy to extract with a regex or parser
  • Length control: "respond in exactly one sentence" or "use at most 3 bullet points, no more"
  • stop_sequences: in the API, set stop_sequences=["</answer>"] to stop generation right after the closing tag — saves tokens
  • No preamble: prefilling with the first word of the expected output removes any intro text entirely

06 — Think Step by Step

Chain of ThoughtReasoning
Core Skill
▾

Asking Claude to reason through a problem before answering — "chain of thought" — dramatically improves accuracy on nuanced tasks like sentiment analysis, multi-step logic, and math. The key is that reasoning must be written out explicitly; Claude cannot think silently.

Why Thinking Helps

Claude, like humans, can jump to confident-sounding answers that are actually wrong. Writing out explicit reasoning catches errors before they appear in the final output. It also counteracts a specific and well-documented Claude bias: when presented with multiple options, Claude tends to favor the last option listed. Explicitly reasoning through all sides before deciding neutralizes this effect.

How to Prompt for Thinking

  • "Think step by step" — the classic, works well for math and logic problems
  • "Before answering, write your reasoning in <thinking> tags" — gives you a visible scratch pad you can inspect
  • "Write the best argument for each side before deciding" — forces balanced consideration on opinion/classification tasks
  • Prefill with <thinking> — forces Claude to start reasoning immediately before it commits to an answer
Without Thinking

Prompt: "Is 'This movie blew my mind with freshness. In unrelated news, I've lived under a rock since 1900.' — positive or negative?"
→ Claude may read surface-level and say positive.

With Thinking

Prompt: "Write the best argument for positive and negative in separate tags, then give your final one-word answer."
→ Claude reasons through the irony and gets it right.

Note: Thinking tokens appear in the output and count toward your token usage. This is standard chain-of-thought prompting — separate from Anthropic's "Extended Thinking" API feature, which uses a dedicated reasoning parameter.

07 — Few-Shot Prompting

ExamplesFew-Shot
Core Skill
▾

Providing 2–5 worked examples (input → output pairs) in your prompt is often more effective than writing detailed instructions. Examples demonstrate format, style, and logic in a way that descriptions alone can't — Claude learns from patterns, not just rules.

Why Examples Beat Instructions

Instructions like "be friendly but concise" are ambiguous. A single example of a friendly-but-concise response removes that ambiguity instantly. Few-shot prompting is especially powerful for: consistent output format, specific writing style, classification tasks, and style transfer.

❌ Instructions Only

"Respond in a warm, parent-like way that encourages imagination without outright lying to the child."
→ Every developer interprets "warm parent-like" differently.

✓ With Examples

Q: Is the tooth fairy real? A: Of course, sweetie. Put your tooth under your pillow tonight…
→ Claude immediately understands the exact tone and approach to replicate.

Few-Shot Best Practices

  • 2–5 examples is the sweet spot — too few loses the pattern, too many wastes tokens
  • Keep examples consistent — if your examples vary in format, the output will too
  • Cover edge cases — include an example that handles the tricky scenario you care most about
  • Use realistic data — synthetic examples that don't reflect real inputs reduce effectiveness
  • Order matters — Claude is most influenced by the last example; put your most representative one last
Works great for: sentiment classification, tone matching, data extraction, Q&A format, and any task where "show don't tell" applies.

08 — Avoiding Hallucinations

AccuracyUncertainty
Quality
▾

Claude will confidently fabricate information when it doesn't know the answer — unless you explicitly give it permission to admit uncertainty. This is one of the most common and costly failure modes in production AI systems. Multiple techniques can be stacked to minimize it.

Why Claude Hallucinates

Claude is trained to be helpful and produce fluent, coherent text. When it lacks specific information, it pattern-matches from related knowledge to fill the gap — and does so with the same confident tone it uses for facts it actually knows. The model has no built-in "I don't know" trigger. You have to install one explicitly.

❌ No constraint

"What was Matterport's exact subscriber base on May 31, 2020?"
→ Claude invents a specific, plausible-sounding number with full confidence.

✓ Constrained

"If you don't know the precise figure, say 'I don't have this information.' Do not guess or approximate."
→ Claude admits it doesn't know.

Anti-Hallucination Techniques

  • "Only answer if you are certain" — explicit permission to abstain when unsure
  • "Say 'I don't have this information' if unsure" — gives Claude a specific phrase so it doesn't have to improvise a refusal
  • Provide the source data — use RAG (see 10.3) to give Claude a document to answer from, eliminating reliance on training memory
  • "Cite where you found this" — hallucinated sources are usually obvious and easy to catch
  • Use XML to isolate the question — wrapping the query in <question> tags signals Claude this is a specific factual query
  • Set temperature=0 — reduces creative variation, which includes creative fabrication
Best practice: Stack multiple techniques together. "Answer only using the context below. If the answer is not in the context, say 'I don't know.' Do not speculate or fill in gaps."

09 — Complex Prompts from Scratch

All TechniquesProduction
Advanced
▾

Production prompts combine every technique from the previous lessons: a detailed role, explicit instructions, few-shot examples, XML-tagged input data, a specified output format, and constraints on uncertainty. The strategy is to start with everything, verify it works, then prune elements that don't add measurable value.

Anatomy of a Complex Prompt

  • 1. Role (system) — who Claude is and its primary goal for this application
  • 2. Task instructions — what to do, stated clearly and step by step
  • 3. Examples (few-shot) — 2–3 input → output pairs demonstrating the expected behavior
  • 4. Input data (XML) — the variable content to process, clearly separated from instructions
  • 5. Output format — exact structure of the response (XML tags, JSON schema, bullet count, length)
  • 6. Constraints — what to do when uncertain, what NOT to include, edge case handling

Build Up, Then Prune

Write a prompt that includes every element above. Test it against real inputs — including edge cases, unusual inputs, and adversarial inputs. Then remove elements one at a time and check whether quality drops. Keep only what earns its place. A simpler prompt is cheaper, faster, and easier to maintain.

Testing Your Prompts

  • Test with edge cases — empty input, very long input, ambiguous input, adversarial input
  • Change one thing at a time — so you know what's actually improving the output
  • Use temperature=0 for testing — ensures reproducible results when comparing prompt versions
  • Check failure modes — what does Claude do when it doesn't know? When the format is violated? When the input is in a different language?
Order matters: Put the role and high-level task at the top of the system prompt. Put the specific data to process at the bottom of the user message, closest to where Claude starts generating. Context near the end has more influence on the output.

10.1 — Chaining Prompts

Multi-TurnRefinement
Appendix
▾

Prompt chaining breaks a complex task into sequential steps, where each Claude response feeds into the next prompt. This lets Claude verify its own work, refine outputs, or handle tasks that are too complex to do reliably in a single pass.

Why Chain Prompts?

A single prompt has limits — for very complex tasks, Claude may make small errors or skip steps. Chaining lets you build a pipeline: Step 1 produces a draft, Step 2 checks it, Step 3 reformats it. Each step is simpler and more reliable than trying to do everything at once.

PythonTwo-step chain
# Step 1: Generate
first_response = get_completion("Name ten words that end with 'ab'.")

# Step 2: Verify — inject Step 1's output as context
messages = [
  {"role": "user",      "content": "Name ten words that end with 'ab'."},
  {"role": "assistant", "content": first_response},
  {"role": "user",      "content": "Check each word above. Do they all actually end with 'ab'? Fix any that don't. If all are correct, return the original list unchanged."}
]

Common Chaining Patterns

  • Generate → Verify — first pass produces content, second pass fact-checks or corrects it
  • Draft → Critique → Revise — Claude writes, then critiques its own work, then improves it
  • Extract → Format — first step pulls raw data from a document, second step structures it as JSON
  • Classify → Respond — first step categorizes the input, second step generates the appropriate response for that category
Safety clause: Always add "if already correct, return the original unchanged" when chaining a verification step. Without it, Claude will sometimes "improve" things that don't need improving.

10.2 — Tool Use (Function Calling)

ToolsFunctions
Appendix
▾

Tool use (function calling) lets Claude interact with external systems — calculators, databases, APIs, search engines. You define the available tools, Claude decides when and how to call them, and you execute the actual code and return the results.

How It Works

Claude doesn't actually run code — it outputs a structured request for a tool call, you detect and execute it, then inject the result back into the conversation. Claude then continues with that result as context. The cycle can repeat multiple times in a single task.

The 3-Step Loop

  • 1. Claude decides to call a tool — outputs a <function_calls> block with the tool name and parameters
  • 2. You execute the tool — halt generation at </function_calls>, run your code, get the result
  • 3. You return the result — inject a <function_results> block in a new user turn; Claude continues
Without Tools

User: "What is 1,984,135 × 9,343,116?"
→ Claude guesses (and is often wrong on large multiplications)

With Calculator Tool

Claude outputs a structured function_call → you execute the multiplication → Claude reads the exact result and responds correctly

Real-world use cases: web search, database queries, sending emails, reading files, checking stock prices, running code, calling any REST API. Tools are how Claude moves from answering questions to taking actions.

10.3 — Search & Retrieval (RAG)

RAGEmbeddings
Appendix
▾

RAG (Retrieval-Augmented Generation) solves a fundamental problem: Claude's training data has a cutoff date, and it can't access your private data. RAG fixes this by fetching relevant documents at query time and inserting them into Claude's context window — so Claude answers from real, current, authoritative information instead of from memory.

Why RAG?

Without RAG, asking Claude about your internal docs, recent events, or niche topics risks hallucination. With RAG, Claude only needs to read and reason over the provided text — a much easier task than retrieving facts from training memory. This dramatically reduces fabrication and makes answers verifiable.

Pattern: Retrieve → Augment → Generate

PythonBasic RAG pattern
docs = search_your_documents(user_query)

PROMPT = f"""Answer using only the information below.

<context>
{docs}
</context>

<question>{user_query}</question>

If the answer is not in the context, say "I don't know"."""

RAG in Practice

  • Embeddings — encode documents and queries as vectors, then find the closest matches by cosine similarity
  • Chunking — split long documents into smaller passages (500–1000 tokens) so the most relevant section can be retrieved
  • Context window — inject retrieved passages into the prompt using <context> tags; Claude reads them, not memorizes them
  • Strict grounding — always include "if the answer is not in the context, say I don't know" to prevent Claude from supplementing retrieved content with hallucinated facts
